## Task 1: Fix Lunch Display Bug + mealType Normalization

**Root cause (from codebase analysis):** `mealType` is stored verbatim in the DB with no normalization. The save API accepts any string. Likely cause of lunch not showing: the log page sends a value that doesn't match what the dashboard's `.toLowerCase()` check expects — e.g. a display label instead of the raw key, or a mismatch between what's sent vs stored.

**Files:**
- Modify: `src/app/api/log/save/route.ts`
- Modify: `src/app/api/dashboard/route.ts`
- Modify: `src/app/(app)/log/[childId]/page.tsx`

**Step 1: Trace the actual mealType value being sent**

In `src/app/(app)/log/[childId]/page.tsx`, find where `mealType` is passed to the save API. Look at the meal type selector state — what values does it use? (e.g. `"Breakfast"` vs `"breakfast"`).

**Step 2: Normalize on save in `src/app/api/log/save/route.ts`**

Add `.toLowerCase().trim()` on the incoming `mealType`:

```typescript
const mealType = (body.mealType as string).toLowerCase().trim();

// Also validate against allowed values
const ALLOWED_MEAL_TYPES = ["breakfast", "lunch", "snack", "dinner"];
if (!ALLOWED_MEAL_TYPES.includes(mealType)) {
  return NextResponse.json(
    { error: `Invalid mealType. Must be one of: ${ALLOWED_MEAL_TYPES.join(", ")}` },
    { status: 400 }
  );
}
```

**Step 3: Fix MEAL_TYPES order in `src/app/api/dashboard/route.ts`**

The constant uses `["breakfast", "lunch", "dinner", "snack"]` but `meal-status.tsx` renders `["breakfast", "lunch", "snack", "dinner"]`. Align them:

```typescript
const MEAL_TYPES = ["breakfast", "lunch", "snack", "dinner"] as const;
```

**Step 4: Verify the fix**

```bash
npm run build
```

Expected: Clean build. Then log a lunch meal in the app and verify it shows the checkmark on Mason's dashboard card.

**Step 5: Commit**

```bash
git add -A && git commit -m "fix: normalize mealType on save, fix meal order in dashboard"
```
