## Architecture Summary

Full constitution: `specs/architecture.md` (208 lines, 8 enforced rules)

**Core invariants the machine must never violate:**
1. All AI calls go through `src/lib/ai/router.ts` — never import providers directly
2. Claude responses always parsed via `extractJSON()` from `providers/utils.ts`
3. Every API route checks `getServerSession(authOptions)` — return 401 if missing
4. Every DB query scopes to `familyId` from the session
5. Route params are async (`{ params: Promise<{ id: string }> }`) in Next.js 16
6. All async operations surface errors to the user — no silent catch blocks
7. Business logic lives in `logic.ts` (pure, testable) — not in `route.ts`
8. Use Prisma-generated types — never redeclare DB model shapes

## Bootstrap Plan

Everything is ready. The machine can start immediately.

```bash
# Run the build machine (implements B4, B5, B7 — 3 features per session)
amplifier recipes execute .dev-machine/recipes/build.yaml

# After a run, verify health
amplifier recipes execute .dev-machine/recipes/verify.yaml

# If build is broken, run the health check to auto-fix
amplifier recipes execute .dev-machine/recipes/health-check.yaml

# After Phase 1 is complete, run QA
amplifier recipes execute .dev-machine/recipes/qa.yaml
```

**Recommended first run order:**
1. B4 (day navigation) — independent, no schema changes
2. B5 (nutrition targets) — new lib file + calculation logic
3. B7 (past meals history) — reuses existing MealList component
4. B8 (ingredient constraints) — requires Prisma migration, commit separately

**To add new features to the backlog:**
1. Write a feature spec using `specs/features/phase1/` as a template
2. Add the spec path to `STATE.yaml → specs.features`
3. The build machine picks it up automatically on next run
